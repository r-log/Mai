# Deploying Mai on a VPS

The board is a single always-on FastAPI app (`mai serve-web`) behind Caddy (which
terminates HTTPS) with SQLite + the git mirrors on local disk. Login is the only
gate; there is no public/anonymous view.

```
internet ──https──> Caddy (:443, auto Let's Encrypt) ──http──> uvicorn 127.0.0.1:8000
                                                                 │ (mai serve-web, systemd)
                                                                 ▼
                                                        /opt/mai/mai.db (SQLite)
                                                        /opt/mai/mirrors (git worker)
```

## Prerequisites (yours)
- A Debian/Ubuntu VPS with a public IP and root/sudo SSH access.
- A domain or subdomain you control (e.g. `mai.r-log.org`) whose DNS **A record** you can point at the box.

## 1. Bootstrap the box
SSH in and run the bundled setup (clones the repo to `/opt/mai`, builds a venv,
generates a strong `SESSION_SECRET` into `/opt/mai/.env`, installs the systemd
unit + Caddy site, starts everything):

```bash
ssh root@YOUR_BOX_IP
curl -fsSL https://raw.githubusercontent.com/r-log/Mai/main/deploy/setup.sh -o setup.sh
sudo MAI_DOMAIN=mai.r-log.org bash setup.sh
```

(Or clone the repo first and run `sudo MAI_DOMAIN=… bash /opt/mai/deploy/setup.sh`.)

## 2. DNS
Point an **A record** for your domain at the box's public IP. Caddy provisions the
TLS cert automatically on first request once DNS resolves.

## 3. Give the board data
Two options:

**a) Ship the already-populated database (fastest — real 242-candidate backlog now):**
From the dev box that produced `mai.db`:
```bash
scp mai.db root@YOUR_BOX_IP:/opt/mai/mai.db
ssh root@YOUR_BOX_IP 'chown mai:mai /opt/mai/mai.db && systemctl restart mai'
```

**b) Harvest on the box (keeps it self-updating, but clones the forks over the network):**
Load a registry of the **GitHub** fork URLs (not local paths), then harvest:
```bash
sudo -u mai bash -c 'cd /opt/mai && .venv/bin/python -m mai.cli registry-load <README-with-fork-urls>'
sudo -u mai bash -c 'cd /opt/mai && .venv/bin/python -m mai.cli commits-harvest && .venv/bin/python -m mai.cli sync-analyze'
```
For periodic refresh, add a cron/systemd-timer running `mai refresh` (the Phase A
backstop); this is the path to wire the GitHub-App webhook accelerator later (Phase C).

## 4. Create accounts
No self-registration — you provision each login and send the one-time password privately:
```bash
sudo -u mai bash -c 'cd /opt/mai && .venv/bin/python -m mai.cli user-add antz --maintainer'
sudo -u mai bash -c 'cd /opt/mai && .venv/bin/python -m mai.cli user-add madmax'
sudo -u mai bash -c 'cd /opt/mai && .venv/bin/python -m mai.cli user-list'
```
Each `user-add` prints a one-time password; the user must set their own on first login.

## 5. Verify
- `https://mai.r-log.org` shows the login form (valid TLS padlock).
- Logging in (then setting a password) reveals the board with the **Port into TWO / THREE**
  columns populated.
- `systemctl status mai` is `active (running)`; `systemctl restart mai` recovers cleanly.
- An unauthenticated visit to any path bounces to `/login` (no anonymous data).

## Notes
- **`SESSION_SECRET`** is auto-generated by `setup.sh` into `/opt/mai/.env` (chmod 600, never committed). `mai serve-web` refuses to start with the default secret while `COOKIE_SECURE=true`.
- **`COOKIE_SECURE=true`** requires HTTPS; Caddy provides it and uvicorn trusts `X-Forwarded-Proto` (`proxy_headers`, `FORWARDED_ALLOW_IPS=127.0.0.1`).
- **Updates:** `cd /opt/mai && sudo -u mai git pull --ff-only && sudo -u mai .venv/bin/pip install -e . && sudo systemctl restart mai`.
- This supersedes the earlier Fly/Pages-oriented `docs/runbooks/phase-a-deploy.md` for the VPS path.
